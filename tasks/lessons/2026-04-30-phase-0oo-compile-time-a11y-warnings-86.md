## Phase 0oo — compile-time a11y warnings (#86) (2026-04-30)

### Insight

- **Reuse the existing `Diagnostic` / `DiagnosticSeverity` types in slot.go; encode the rule code as a `"code: message"` prefix on `Message`.** First draft introduced parallel `Severity` and `Diagnostic` types in `a11y.go` and immediately collided with the existing decls in `slot.go` (which is FORBIDDEN to edit this phase). The clean fix is to consume the existing surface as-is and put the rule taxonomy on the `Message` field — no struct extension, no parallel types, no edit on a forbidden file. A future cross-cutting refactor can hoist `Code` into the struct when it's worth touching slot.go.
- **`testdata/codegen/<rule>/` is a trap; `TestFixtures` in the FORBIDDEN `codegen_test.go` walks that tree and tries to render every `.svelte` as a page template.** Phase 0bb already documented this for component fixtures (lesson 7); reconfirmed here. New generators (a11y, components, layouts, error pages) get sibling subtrees under `testdata/<kind>/` with their own walker in their own `_test.go`. The prompt's suggested path conflicted with this — convention won.
- **`hasMeaningfulChild` for accessible-name checks must walk into blocks (`{#if}`, `{#each}`, `{#await}`, `{#key}`), not only direct text/element children.** A naive "look at immediate children" implementation flags `<button>{#if active}Save{:else}Saving{/if}</button>` as missing content, which is wrong. The recursive walker treats any branch with non-whitespace text or a Mustache as "meaningful" — same liberal rule Svelte's a11y plugin uses. False negatives (every branch is whitespace) are acceptable; false positives on real templates are not.
- **`role` attribute validation skips dynamic and interpolated values by design.** The check matches against a static role token list; a dynamic expression (`role={Dynamic}`) is by definition unknowable at codegen time. Refusing to flag is the right call — the alternative is "validate at runtime", which is out of the codegen pass's mandate.
- **Input-label association is over-permissive on purpose: `id` alone satisfies the rule.** Codegen has no view of sibling `<label for="...">` elements — they may live in a parent layout or be wired via a prop. Treating `id` presence as "labeled" matches Svelte's a11y plugin's behavior and avoids false positives at the cost of not catching `<input id="x">` with no matching `<label for="x">` anywhere. The trade matches the project's "warnings are advisory" policy.

### Self-rules

1. **Before introducing a new exported type in a package with existing diagnostic infrastructure, grep the package for `Diagnostic`/`Severity`/`Error` decls.** Reuse the existing surface unless extension is impossible without it. Parallel types fork the audit surface and break in collision-prone places (forbidden files, golden tests, lint rules).
2. **A new generator's fixtures live OUTSIDE `testdata/codegen/`.** Use `testdata/<kind>/` (e.g. `testdata/a11y/`, `testdata/component/`). The `codegen/` tree is owned by `TestFixtures`; never extend that walker's skip list unless you own `codegen_test.go` this phase.
3. **Accessible-name checks (`<a>`, `<button>`, label) recurse through block constructs, not only direct children.** Walk `{#if}`/`{#each}`/`{#await}`/`{#key}` branches the same way the codegen emit pass does. Naive immediate-child walks produce false positives on every real template.
4. **Validation rules over static taxonomies (ARIA roles, HTML elements, etc.) skip dynamic and interpolated attribute values.** A codegen-time check cannot know runtime values; flagging them produces noise. Static-only validation is the intentional shape, not a bug.
5. **`a11y` warnings are non-fatal by default.** Do NOT wire into `b.Fail`. Return `[]Diagnostic` for the caller (CLI/build) to render; promotion to fatal lives behind an explicit flag (deferred follow-up). The same standalone-callable shape lets editor integrations (`packages/lsp/`) consume the same pass.
