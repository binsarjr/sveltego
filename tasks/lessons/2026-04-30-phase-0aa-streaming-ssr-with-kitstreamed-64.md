## Phase 0aa — streaming SSR with kit.Streamed (#64) (2026-04-30)

### Insight

- **A streaming render path can't reuse the buffered Response carrier.** The existing pipeline returned `*kit.Response{Body: []byte}` and let `writeResponse` fan it out at the end. Streaming flips the dependency: the render path needs the live `http.ResponseWriter` to call `Flush` between chunks. The clean fix mirrors the existing `+server.go` short-circuit — emit a sentinel error (`errStreamingWrote`) that the outer `handle()` matches with `errors.Is` and returns without further writes. Sentinels are honest about "this code already wrote the response, don't double-write" in a way a nullable Response field can't.
- **Reflection over the user's PageData is the only deferral-friendly registration mechanism.** Forcing the user to call `ev.RegisterStream(s)` puts the goroutine lifecycle on the user's plate; making `Stream(fn)` synchronous defeats the purpose. The chosen shape — `Stream(fn)` spawns the goroutine immediately, the pipeline walks the typed PageData via `reflect.Value.Field(i)` looking for `kit.StreamedAny` implementers — keeps the user surface to "return a value" while the runtime owns timeouts, cancellation, and emission order. Cost: top-level fields only (no nested traversal). That's a deliberate trade — nested streaming is a nice-to-have that rewards over-engineering before there's a user.
- **Generic types satisfy interfaces through method sets, not the type parameter.** `*Streamed[T any]` with methods `StreamID() uint64` and `WaitAny(ctx, timeout) (any, error)` satisfies the package-level `StreamedAny` interface for every T without a per-instantiation registration. The pipeline's reflection walk only needs the interface check; it never inspects T. This is the same pattern Go's stdlib uses for `sync.Map` consumers — the type-erased interface is the runtime contract, the generic struct is the user contract.
- **`json.Marshal` does not HTML-escape; `json.Encoder.SetEscapeHTML(true)` does.** A naive `<script>__sveltego__resolve(id, JSON)</script>` emission with `json.Marshal` is XSS-vulnerable when the streamed value contains `</script>`. The fix is targeted: post-process the encoded bytes to replace `</` and `<!--` with their `<` escapes. Other `<` characters can stay raw because they're inside a JSON string literal, which the JS parser treats as ordinary text. Don't reach for the encoder — its HTML mode escapes more aggressively than needed and breaks structural characters (`>` inside arrays). The narrow escape function is auditable in 8 lines.
- **Goroutine cleanup is the user's job; the runtime documents the contract.** `Stream(fn)` does not pass a `ctx` into `fn` — it's the user's closure, the user captures whatever context they need (typically the LoadCtx's request context). The pipeline's `Wait(ctx, timeout)` honors the request context for the *waiter* side, not the producer. Conflating the two would force `Stream` to grow a context parameter that 90% of users would copy-paste from a tutorial without thinking. Document the convention; let `select { case <-ctx.Done(): }` inside fn be the user's explicit opt-in.
- **Flush after every emission, not at the end of the loop.** A buffered `for _, s := range streams { emit(s); collected = append(collected, payload) }` followed by one `FlushTo(w)` defeats streaming entirely — the client sees one big write at the end. The flush must be per-iteration so each `<script>__sveltego__resolve(...)` chunk hits the wire as soon as its producer goroutine completes. Verify with a test that holds one stream blocked while another resolves and asserts byte-order at the client.

### Self-rules

1. **Sentinel errors are the right channel for "the inner handler already wrote the response".** Match `errServerRouteWrote`'s shape: a package-private `errors.New` value, an `errors.Is` check in the outer `handle()`, no further writes after the match. New paths that take ownership of the `http.ResponseWriter` (streaming, websocket upgrade in user code, etc.) should mint a new sentinel rather than overload an existing one — `errors.Is` reads as documentation when the names are distinct.
2. **Reflection-driven discovery walks top-level struct fields only.** Recursive descent into maps, slices, and pointers tempts callers into nested-stream patterns whose error semantics are undefined ("which stream's timeout fires when the parent struct is nil?"). Keep the walk shallow, document the constraint in the godoc, and let users hoist nested streams to the top level if they want them registered.
3. **Type-erased interfaces for generic types live in the same package as the generic type.** `StreamedAny` next to `Streamed[T]` keeps the contract greppable and lets the type-parameter-free interface stay an internal-ish API even though it's exported. Don't put the interface in the consumer package (the pipeline) — that's where it gets called, not where it gets defined. Same convention as `io.Reader` living in `io`, not in every reader's caller.
4. **HTML/script escapes for inline JSON neutralize "</" and "<!--" only, not all of "<".** The narrow rule is faster, easier to audit, and doesn't break structural characters. The standard quote — Go's `text/template` html.go — does the same dance. Avoid `json.Encoder.SetEscapeHTML(true)` for inline-script payloads because it mangles `>` and `&` inside arrays/strings, producing valid JSON but brittle reading inside `<script>`.
5. **`Flush` between chunks, not at the end.** `bytes.Buffer`-style accumulation with one terminal flush is the buffered render's pattern; streaming render flips it. Per-chunk flush is the load-bearing behavior — verify with a test that asserts byte-order on the client side, not just final-body content. The test must hold one stream blocked while another resolves so the assertion fails when flush is misplaced.
6. **The `ctx` for the producer goroutine is the user's closure responsibility; the runtime owns the waiter context.** `Stream(fn)` deliberately accepts `func() (T, error)` not `func(context.Context) (T, error)` — adding the context parameter would imply the runtime cancels the producer on request abort, which it can't (Go has no goroutine kill). The runtime cancels the *waiter* via `Wait(ctx, timeout)`; the producer either finishes naturally or leaks. Document the leak risk; let user code wire `<-ctx.Done()` inside fn when it matters.
7. **Streaming integration tests assert byte-order, not just byte-presence.** "the body contains both the shell and the payload" passes for buffered AND streaming responses. The assertion that distinguishes them is "the shell can be read before the payload-blocking goroutine releases" — a `bufio.Reader.Read` with a deadline before the release-channel send catches a misplaced flush that a `ioutil.ReadAll`-style test would miss.
