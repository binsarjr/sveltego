## Phase 0t — error boundary rendering (#28) (2026-04-30)

### Insight

- **Boundary metadata belongs on the matched route, not on the event.** `handlePipelineError` runs after the inner pipeline blew up, so it cannot re-walk the route tree to discover the +error.svelte chain. The natural fix is a small back-channel: a `*router.Route` pointer captured by a closure inside `handle()` that the resolve closure stamps on a successful match. Threading this through `respondWithError(... route *router.Route, ...)` keeps the rest of the error path untouched and avoids leaking router types into kit.
- **Outer-vs-inner is a depth count, not a slice copy.** `ErrorBoundaryLayoutDepth = layoutsAtOrAbove(chain, boundaryDir)`. The runtime composes `LayoutChain[:depth]` outer→inner around `route.Error(...)`, no allocation, no parallel "errorChain" abstraction. This reuses the layout composition pattern that Phase 0k-A landed; the only new structural piece is the `Error ErrorHandler` field.
- **A directory holding only +error.svelte still needs a generated package.** routescan already promotes any dir with a special file into a ScannedRoute, so `route.HasError && !route.HasPage && !route.HasServer` is reachable. Codegen must emit `error.gen.go` for that dir even though manifest.go skips it from the `entries` slice — the manifest's import block needs the package alias regardless. Tracking error imports separately (next to layoutImports) keeps the alias reservation logic uniform.
- **Error template binds `data` to `kit.SafeError` directly, no wrapper struct.** Issue #28's example wrote `ErrorData{Error: safe}`; the simpler shape uses kit.SafeError as the parameter type and lets users write `{data.Code}` / `{data.Message}` / `{data.ID}`. One less layer of indirection, one fewer alias to maintain in `+error.svelte`. The codegen emits `func (e ErrorPage) Render(w *render.Writer, ctx *kit.RenderCtx, data kit.SafeError) error` so the template lowering reuses the existing mustache/element pipeline unchanged.
- **Recursive boundary failure must drop back to plain text, not re-enter the boundary.** When `route.Error(...)` itself returns an error during render, calling HandleError again risks an infinite loop if the SafeError shape is what the boundary chokes on. `respondWithError` short-circuits to `writeSafeError` (the Phase 0h plain-text writer) on failure and logs the boundary error once.
- **Manifest emit conditionals must cover all boolean combinations of feature flags.** Pre-0t the import block only branched on `hasPage || hasLayout` for `kit`/`render`/`fmt`. A route table with only error packages (no page, no layout) needed `kit` and `render` but not `fmt`. The fix is a `switch` block ordering page/layout (with `fmt`), then error (without `fmt`), then options-only (kit alone), then default. Generated source stays minimal but compiles for every shape of input scan.

### Self-rules

1. **When the pipeline needs the matched route after the resolve closure errors, capture it via a pointer-to-pointer at the `handle()` scope.** The closure stamps `*matched = route` on a successful match; the error path reads it through the captured pointer. Avoid threading the route into kit.RequestEvent — that would surface a router-internal type to user code, and Locals as a stash bag is too easy to overwrite from a Handle hook.
2. **Reuse `LayoutChain[:depth]` slicing for any feature that says "render only the layouts above some boundary".** A new error-chain abstraction would duplicate the composition loop and require parallel emission in the manifest. The only metadata the runtime needs is `(handler, depth)`. Keep the chain authoritative; new boundaries are slices of it, never replacements.
3. **Codegen metadata for "covers this route" relationships propagates to every leaf during scan, not synthesized at emit time.** `ScannedRoute.ErrorBoundaryDir` / `ErrorBoundaryPackagePath` / `ErrorBoundaryLayoutDepth` is computed once during routescan when the directory tree is in hand. The manifest emitter then reads three flat fields per route — no second tree walk, no per-emission ancestor lookup. Same pattern as `LayoutPackagePaths`.
4. **For a "render this kit type directly" template, don't introduce a per-template DataType alias.** The error-page lowering binds `data kit.SafeError` because `kit.SafeError` is already exported and stable. Adding `type ErrorData = struct{...}` would force users into a synthetic field name (`{data.Error.Message}` vs `{data.Message}`) for zero gain. Reserve the alias dance for when codegen needs to splice user-declared fields into the parameter type (PageData, LayoutData).
5. **Recursive failure modes get an explicit fallback, not a panic-recover loop.** When the boundary itself fails, the response path drops to the simplest writer the framework has (plain text). Logging the boundary error once at Error level lets ops see it; a user can debug by removing the +error.svelte temporarily. Never retry an HTML render with a "simpler" boundary template — the failing template might be the only one the user owns.
6. **Manifest import emission uses a switch over feature presence, not chained `if` clauses.** As features add (pages, layouts, errors, options-only), the import block needs every combination of `kit`/`render`/`fmt` resolved correctly. A switch makes the precedence obvious at a glance: page/layout owns the broadest import set (incl. fmt for type-mismatch errors), error owns kit+render, options-only owns kit alone. Stays gofumpt-clean and reviewer-comprehensible.
