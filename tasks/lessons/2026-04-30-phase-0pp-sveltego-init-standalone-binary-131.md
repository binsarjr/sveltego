## Phase 0pp — sveltego-init standalone binary (#131) (2026-04-30)

### Insight

- **`sveltego-init` belongs at `packages/init/` per the LSP/MCP precedent — not `cmd/sveltego-init/` at repo root.** `packages/lsp` and `packages/mcp` already established the standalone-binary convention with their own `go.mod`, `cmd/<tool>/`, `internal/<scope>/`. Following the precedent keeps release-please per-package versioning, the `go.work` `use` block, and the workspace lint/test loop wired with zero CI surgery. The issue body's `cmd/sveltego/init.go` snippet was a sketch, not a layout instruction.
- **A separate `go.mod` per scaffolder keeps init-only deps out of the main runtime.** The init binary depends on `templates/ai`'s `embed.FS` and nothing else; pulling it into `packages/sveltego/go.mod` would broaden the surface every time scaffold gains a flag. The `replace` directive points the AI templates dep at the in-repo path so workspace+isolated-mode both build clean.
- **TTY detection by `os.ModeCharDevice` over `*os.File.Stat()` is enough for piped stdin.** No `golang.org/x/term` dep, no `unix.IsAtty` cgo. Test piping `echo n |` flips the bit; the binary defaults `--ai` to false on non-TTY without ever prompting. Keeps the dep tree at zero non-stdlib imports for the CLI.
- **golangci-lint's `gosec` rule G306 forces `0o600` on `os.WriteFile`; the project already standardized on this via `genFileMode = 0o600` in codegen.** Local repo convention beats POSIX-default `0o644`; the lint catches drift and CI fails on `0o644` even for write-once scaffold output. Reuse the same constant pattern (`fileMode = 0o600`) so the next file-emitter scaffold lands lint-clean on the first push.
- **The `flag` stdlib's `--help` exits non-zero by design (`flag: help requested`).** Test harnesses that grep for "exit 0 on --help" trip on this. Either accept the exit code or roll a custom usage handler. For a scaffolder where `--help` is informational, the stdlib behavior is fine and saves a custom flag set.

### Self-rules

1. **A new standalone CLI tool gets its own package directory under `packages/`, mirroring `lsp/` and `mcp/`: own `go.mod`, `cmd/<tool>/`, `internal/<scope>/`, `README.md`.** Don't graft another `cmd/` tree at repo root. Workspace + CI fan-out + release-please all key off the per-package layout. The issue body's filename hints (`cmd/sveltego/init.go`) describe the integration point inside an existing binary, not the layout for a new one.
2. **Init/scaffold-style tools depend on `templates/ai` (or any future `templates/*`) via a `replace` directive in their own `go.mod`.** Keeps templates out of the runtime dep graph. Tests assert byte-equality against the `embed.FS` so drift between scaffolder output and the embed source surfaces immediately.
3. **Use `(*os.File).Stat().Mode()&os.ModeCharDevice != 0` to detect TTY in CLI tools.** No `x/term`, no cgo. Wrap it in an `isTerminal(io.Reader)` helper that returns false for non-`*os.File` so test injection (bytes.Buffer) routes to the non-interactive branch automatically.
4. **Reject overwrites by default; surface skipped paths in stdout; require `--force` to overwrite.** The pattern is: `os.Stat` first, append to `Result.Skipped`, return early. Single source of truth for the conflict policy is the `writeFile` helper — never duplicate the `force` check at call sites.
5. **`os.WriteFile` in non-test code uses `0o600` per repo convention (`genFileMode` in codegen, now `fileMode` in init). Tests with seeded fixture files may use `0o644` because the `_test.go` exclusion in `.golangci.yml` disables `gosec` there.** Keep one constant per package; never spell `0o600` literal at call sites.
6. **`stdlib flag --help` exits non-zero. Document this in the README/build-doc when the user verifies via `--help`.** Don't paper over with a custom error handler; stdlib behavior is the convention. If a CI gate must check `--help` works, redirect stderr and grep for the usage banner instead of asserting exit zero.
