## Phase 0mm — bench suite (#60) (2026-04-30)

### Insight

- **`bench/` and `benchmarks/` are sibling workspace modules with disjoint scopes.** `benchmarks/` (RFC #105) gates per-package micro-benches with the existing `bench-compare` tool; `bench/` (#60) gates full-pipeline SSR scenarios via the same tool. Two modules, two suites, one comparison binary. First-pass design kept `bench/` outside `go.work` to "isolate driver deps" — but the repo's pre-commit hook iterates `go.work` modules to lint and `go test`s changed packages from the workspace root, both of which fail on out-of-workspace members. The cleaner design adds `bench/` to `go.work` next to `benchmarks/` (it carries no extra deps beyond what `packages/sveltego` already pulls in) and keeps both suites visible to the same toolchain.
- **Workflow gates two suites with one `bench-compare` invocation each, plus one combined exit check at the end.** First-pass design tried to merge both `go test -bench` outputs into one file; benchstat correctly rejected the merge because the package paths differ. Splitting into two `/tmp/base*.txt` + two `/tmp/head*.txt` pairs and chaining the exit codes (`rc1 || rc2`) keeps each suite's diff readable in CI logs and lets either gate fail independently. The "if grep BenchmarkServeHTTP on baseline" guard handles the bootstrap commit cleanly: when `bench/` did not exist on `HEAD~1`, the regression gate degrades to a no-op for that one run rather than failing on absent fixtures.
- **`oha`/`wrk`/Bun comparison is documented and deferred, not stubbed.** Issue #60's headline goal compares sveltego against `@sveltejs/adapter-bun`. That needs Bun in CI, a parallel SvelteKit app per scenario, and `oha` on PATH. None of those land in MVP scope without ballooning the runner image. The shipped harness uses `httptest` (zero external deps) for the sveltego-vs-sveltego regression gate; the cross-runtime comparison is filed as a v1.0 hardening deferral with a placeholder script (`scripts/adapter-bun-compare.sh`) that prints a deferral notice and exits 0. Reviewers see the deferral in the PR body, the README, and the script itself — three surfaces, no surprise.
- **Scenarios stay Go-only; no `.svelte` source under `bench/`.** The instruction sketched `scenarios/*.svelte` template fixtures alongside `scenarios/*.go`. Implementing real `.svelte` fixtures would require running codegen against `bench/` as part of every benchmark prep step, doubling the maintenance surface and tying the benchmark numbers to codegen-pipeline overhead instead of pure runtime performance. The shipped scenarios use direct `PageHandler` closures that mimic what codegen emits — same signature, same allocation profile, no codegen pre-step. When codegen-output benchmarking matters (it doesn't yet for #60's MVP slice), it gets its own scenario kind under `bench/scenarios/codegen/`.
- **The driver CLI (`cmd/sveltego-bench`) is a convenience, not the gate.** Issue #60 lists `oha`/`wrk` as load tools; the gate uses `go test -bench=.` because that's what `bench-compare` understands. The driver exists for human runs ("show me rps and p99 for one scenario right now") and produces tab-separated output that's easy to paste into PR bodies. It is NOT wired into CI — duplicating measurement paths invites two sources of truth for the same number.

### Self-rules

1. **A new bench dir at repo root is a `go.work` member, never an isolated module.** The pre-commit hook iterates `go.work` to lint changed packages and `go test`s them from the workspace root; out-of-workspace members blow up the hook with "directory contained in a module that is not one of the workspace modules". Add the dir to `go.work` the moment its `go.mod` lands. Reserve a separate module-graph (replace directives, future Bun harness deps) for the day it actually carries deps the main workspace must not see.
2. **When extending a CI workflow that runs two suite kinds, gate them as siblings — separate base/head files, separate `bench-compare` invocations, single combined exit at the end.** Merging suite outputs into one file breaks benchstat's package-grouping logic and obscures which suite regressed in the log output. The chained-exit pattern (`rc1 || rc2`) keeps both visible.
3. **When the bench dir is absent on the diff base (bootstrap commit), guard with a `grep -q BenchmarkX on baseline` check that degrades the gate to a no-op rather than failing on absent fixtures.** First-run gates fail in confusing ways otherwise — the bench step appears red even though the suite is brand new and has nothing to compare against. The grep-guard is one line and the failure mode is obvious from the printed report.
4. **Cross-runtime comparison harnesses (Bun, Node, Deno) are deferred to a follow-up unless the runtime is already on the CI image.** Adding a runtime to CI for one bench step is a 10x scope creep. Document the deferral in three places — PR body, package README, placeholder script — so the gap is visible without grep.
5. **Bench scenarios use direct handler closures, not `.svelte` source, until codegen-output benchmarking explicitly matters.** The codegen pipeline has its own per-package guard in `benchmarks/`; doubling that exercise inside `bench/` couples the SSR-perf gate to codegen-perf changes. Keep scenario shapes as Go closures that mirror codegen output; introduce a `scenarios/codegen/` subtree only when the issue asks for it.
6. **Driver CLIs alongside `go test -bench` benchmarks are a convenience, never the regression gate.** `bench-compare` consumes `go test -bench=.` text; the gate stays on that path. The CLI is for humans wanting rps/p99 in a hurry. Do not wire CLI output into CI even when the format is similar — two measurement paths for the same number is two sources of drift.
