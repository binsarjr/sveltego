## Phase 0v — codegen template features ({@html} + {@const}) (#55 + #56) (2026-04-30)

### Insight

- **The lexer / parser / AST / codegen dispatch for `{@html}` and `{@const}` was already wired across earlier phases.** What looked like greenfield (issues #55 and #56) turned out to be a position-rule + golden-coverage gap: `RawHTML` and `Const` AST nodes already flow through `emitNode`'s switch since the `@`-tag tokenizer landed; `WriteRaw` lived in `render.Writer` since the writer pass. The work shrank to (a) enforcing the spec'd "no `{@const}` at template root" diagnostic, (b) making the existing `39-at-const.svelte` fixture comply, and (c) adding multi-const + each-block fixtures the previous coverage missed. Always grep AST nodes and codegen dispatch before assuming a feature isn't started — upstream phases may have shipped 80% of the lowering.
- **Position rules belong on a root-walker pre-pass, not on `emitNode`.** Threading an `inBlock bool` through every block emitter is the naïve shape. The cleaner one: a single `rejectRootConst(b, frag.Children)` called from `Generate` and `GenerateLayout` that latches the diagnostic via `Builder.Fail`. `emitNode` stays oblivious; block emitters keep emitting verbatim because they are nested by construction. The pre-pass is one slice scan and the diagnostic carries the original `ast.Pos` so the CLI can render caret context.
- **Existing fixtures sometimes encode behavior that the spec later forbids.** `39-at-const.svelte` placed `{@const}` at root and the prior golden treated that as valid. The spec says root-level `{@const}` must be a build error. Resolution: edit the fixture to wrap the const in `{#if Show}`, regenerate the golden, add a parallel root-level fixture (`64-at-const-at-root.svelte`) plus error golden so the diagnostic is locked. Never delete a fixture to land a stricter rule — promote it into the inside-block coverage and add a parallel error fixture.
- **`assertErrorGolden` truncates only on stable framework prefixes.** The helper preserves the `line:col` prefix and trims after `: invalid Go expression` or `: invalid Go statement` so `go/parser` wording drift doesn't flake CI. New error categories (here: `{@const} not allowed at template root`) bypass the trimming and store the full message — fine because the message is framework-owned and stable. If a third truncation prefix lands later, extend the trim list rather than silently mixing full and trimmed forms.

### Self-rules

1. **Before writing code for an open issue, grep the codebase for the AST node names and codegen dispatch entries.** Run `grep -rn '<NodeType>\|@<keyword>'` across `internal/ast`, `internal/parser`, `internal/codegen` first. A feature frequently has 80% lowering already shipped from a prior phase; the issue scope often shrinks to validation rules and golden coverage.
2. **Position-rule diagnostics live in a single pre-pass over `frag.Children`, not in a flag threaded through every block emitter.** The pre-pass reads the slice once, latches a `CodegenError` via `Builder.Fail`, and returns. Block emitters stay agnostic about whether they're at root because by construction they cannot be (their parent is the block AST node). One source of truth for the rule, no flag plumbing.
3. **When a spec change invalidates an existing fixture, edit the fixture to satisfy the new rule and add a parallel rejection fixture.** Deleting coverage to land a stricter rule loses the success-path golden; moving its content into the surrounding-block form preserves it. The new error fixture pins the diagnostic so a regression flips both a unit test and a golden mismatch.
4. **Root-level `Generate` and `GenerateLayout` are the only two call sites that need the position-rule pre-pass.** Snippet bodies, layout chains, and other future surfaces inherit the rule by composition: their child fragments live inside a block boundary. Adding more call sites means duplicating the call; refactor only when the third site appears.
