## Phase 0x — AI assistant templates (#73) (2026-04-30)

### Insight

- **Issue body and current ADR amendments disagreed on user `.go` filenames.** The #73 spec referenced `+page.server.go`; ADR 0003 (Phase 0i-fix) settled on `page.server.go` (no plus) plus `//go:build sveltego`. Templates must teach the **current** convention, not the issue body. Followed the same pattern as #29 (Phase 0r): build against the actual scanner contract, note the divergence in the lessons log so the next reader of #73 sees the convention was already corrected at codegen time.
- **`go:embed` cannot reach files via `..`.** That forced a choice: either copy the templates into a Go-module dir (drift risk) or make `templates/ai/` its own module. Chose the second: `templates/ai/go.mod` makes `github.com/binsarjr/sveltego/templates/ai` importable and keeps the markdown next to the `embed.FS` declaration. The future `init --ai` (filed as a follow-up issue) imports this module instead of duplicating the bytes.
- **Snapshot-as-byte-equality would punish every legitimate edit.** Templates ship as content; routine wording fixes are expected. A SHA256 fixture would force a `-update` dance for every comma. Replaced the byte-snapshot approach with positive-anchor assertions: each template must contain canonical substrings (`//go:build sveltego`, `kit.Redirect`, `$props`, `https://sveltego.dev`, etc.) and must cross-link to its siblings. Drift on the **load-bearing content** breaks the test; cosmetic edits don't. Anti-pattern coverage uses the same idea — assert the rejected token is **named** so anyone grepping for it lands in a rejection context.
- **Initial anti-pattern detector was a context-window heuristic. Wrong.** First version scanned for `+page.server.go` / `export let` and required "Don't" / "rejects" within ~120 chars. The Don't section's heading sits more than 120 chars above the bullet on long files; the heuristic flagged legitimate rejection bullets as violations. Fix: invert the test. Assert each rejected token **appears** at least once (positive coverage); writers can't accidentally drop the rejection bullet without the test catching it. Simpler, no false positives.

### Self-rules

1. **When a feature ships content that lives outside any existing Go module, give it its own `go.mod` rather than copying bytes into a sibling package.** `embed` cannot escape its module root; dual-source content drifts the moment one copy gets edited. A standalone module costs three lines (`go.mod` + `go.work` entry + import path) and keeps the content authoritative. Reserve cross-module copying for the rare case where the consumer cannot add a dependency (build-time bootstrap, vendored snapshot for offline CI).
2. **Snapshot tests for content that wants to evolve assert anchors, not byte-equality.** Pick the load-bearing strings (URLs, function names, build constraints, cross-links) and require their presence. The test fails when meaning drifts; cosmetic edits stay friction-free. Reserve byte-exact snapshots for outputs that **must** be deterministic (codegen goldens), not for human-edited prose.
3. **Anti-pattern coverage is a positive assertion, not a window heuristic.** "Banned token must not appear without rejection context" is brittle (the rejection heading may be one section away). Flip it: "rejected token must appear at least once" — guarantees the rejection bullet exists, runs in O(n) over each file, no false positives.
4. **When the issue body and a later ADR amendment disagree, the ADR wins; carry the divergence note in the PR description.** Same rule as Phase 0r (#29) — the ADR is the live source of truth, the issue body is a historical spec snapshot. Never edit a closed RFC issue body to retroactively match the amendment; the divergence note in the PR keeps the chain visible.
5. **`init --ai` flag without an `init` command is a stub waiting to break.** sveltego's CLI ships `build`/`compile`/`dev`/`check`/`version` only; no `init`. Faking an `init` purely to attach `--ai` would land a half-feature. The clean path: ship the templates package, file a follow-up issue for the CLI integration once `init` itself lands. Half-finished features are a "no over-engineering" violation regardless of how small the gap looks.
