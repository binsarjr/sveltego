## Phase 0z — Svelte 5 runes ($props/$state/$derived/$effect/$bindable) (#43-#47) (2026-04-30)

### Insight

- **Hand-roll the rune line classifier; do not coerce go/parser into accepting Svelte syntax.** First attempt wrapped loose top-level statements inside a synthetic `func __sveltegoTopLevelStmts__()` so go/parser would consume the rune lines as statements, then reverse-engineered the destructure pattern from the resulting AST. Working but fragile: every shape variant (type annotation, rest spread, bindable default) required a different post-parse recovery rule, and the synthetic function shape leaked into error messages. Rewrote as a line-by-line classifier that recognizes `let ` and `__sveltegoRune__effect` prefixes, joins continuation lines until paren-balance settles, and parses only the RHS through `parser.ParseExpr`. The shape of each rune is small and well-defined; matching it directly costs ~50 LOC vs ~150 LOC of post-parse archaeology.
- **`$props()` on a top-level page is a component-internal contract, not a function parameter.** Issue #43's example shows `func Render(props Props, w *render.Writer)`, but `+page.svelte` Render signatures are owned by the runtime — `(w, ctx, data PageData)`. Adding a `props Props` parameter would require touching the manifest/router/server/runtime, all of which are explicitly out of scope for this phase. The clean compromise: emit `Props` struct + `defaultProps` helper at package scope, then declare `var props Props; defaultProps(&props)` at the top of Render's body. Templates reference `props.X` as the issue intends; the parent-passes-props story lands when component nesting (#29) does, at which point the Render signature can grow without breaking existing routes.
- **Bindable is a `$props()` field with a tag, not a separate construct.** Issue #47 wants two-way binding metadata exposed to the client codegen pass (#34). Encoding bindable-ness as a struct tag (`kit:"bindable"`) keeps the metadata adjacent to the field rather than in a parallel map; client codegen reads the tag via reflect when it lands. No `kit.Bindable[T]` wrapper type, no metadata table — the tag IS the metadata. Saves a public API surface that the v0.4 milestone hasn't finalized.
- **`$effect` on the server is empty, full stop.** Tempting to emit a marker comment, a hydration anchor, or an `_ = effectSlot` placeholder so future passes have a hook. None of those have a consumer yet, and the issue explicitly says "remove from server emit". Emitting `// $effect elided on SSR` keeps the diff visible without producing runtime cost; emitting nothing would be cleaner still but loses the documentation breadcrumb when a developer is debugging "why isn't my effect firing on the server".
- **`$derived.by` IIFE shape requires explicit parens around the function literal.** `func() T { ... }()` parses but `format.Source` reformats it as `func() T { ... }()` — fine until the body has comments or the receiver carries a `func(...) T { ... }` shape that go/printer reflows differently. Wrapping in parens (`(func() T { ... })()`) makes the precedence explicit and survives every formatter rule. Three extra characters; zero edge cases.

### Self-rules

1. **When a rune call has a well-defined shape, parse the shape directly via line-classifier + `parser.ParseExpr` on the RHS — do not coerce the source into a Go-parseable shape and recover meaning post-parse.** Coercion-then-recovery requires writing two parsers (the coercer and the recoverer) and they will disagree on edge cases. A direct line classifier (`HasPrefix("let ")` + paren-balance join + `ParseExpr`) is one pass and one parser.
2. **When a feature in a downstream-owned signature is "out of scope, but the issue example shows it", emit the feature at package scope and consume it inside the existing signature.** `Props` struct + `defaultProps` at package scope, `var props Props` at the top of Render's body. Template references `props.X` as the issue intends without touching the runtime/manifest contract. Re-evaluate when the dependent issue (#29 component nesting) lands.
3. **Metadata for client-side wiring lives on the field via struct tags, not in parallel maps.** `kit:"bindable"` ships with the Props struct itself; the client codegen pass reads it via `reflect.StructTag.Get`. A `bindableFields = map[string]bool{...}` table would have to be regenerated and kept in sync; the tag is the source of truth.
4. **Loose top-level statements that are NOT recognized rune calls remain top-level declarations and fail with the existing "expected declaration" error.** The line classifier only routes lines beginning with `let ` (containing rune tokens) or `$effect`-prefixed calls. Bare assignments (`x := 1`) sit on the residue path and surface the same diagnostic as before. Don't expand the classifier to "any statement"; that's a different issue, and it needs the runtime concept of a render-body local scope to be useful.
5. **Server-side rune lowering is one-shot evaluation. SSR has no notion of reactivity, no state-tracking, no cleanup callbacks.** `$state(x)` becomes `name := x`; `$derived(expr)` becomes `name := expr`; `$effect(fn)` becomes nothing. Document this in the codegen comment block so reviewers don't ask "where does the reactivity go?" — it goes to the client bundle, which Vite handles independently.
