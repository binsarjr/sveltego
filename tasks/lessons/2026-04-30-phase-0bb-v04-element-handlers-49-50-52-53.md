## Phase 0bb — v0.4 element handlers (#49 + #50 + #52 + #53) (2026-04-30)

### Insight

- **Closure-based slot lowering is the same primitive as the existing layout `children` parameter — just per-name and per-prop.** Layouts already pass `children func(*render.Writer) error` as the last param to `Render`; the natural extension for components is a `Slots` struct whose fields are typed closures. No new runtime registry, no string-keyed lookup, no separate "slot system". `<slot name="header"/>` lowers to `slots.Header(w)`; `<slot item={item}/>` to `slots.Default(item, w)`. Same code path on both sides.
- **`<svelte:component this={X}>` does not need a per-site interface generator.** The issue body proposed a full adapter machinery (per-site `pageComponent` interface, generated wrapper structs per candidate). For the SSR-only lowering Phase 0bb landed, that's overkill: the user's expression must already be a value that exposes `Render(*render.Writer) error`, so codegen just emits `expr.Render(w)`. Any "static set" optimization (switching on a discriminator) is the user's switch in their own Go code — codegen takes the result. Less generated code, no synthetic interface names, and the receiver shape stays identical to `Component.Render(w)` already used for layouts.
- **Position validation works through a single `Builder.nestDepth` counter, not a separate AST walk.** `<svelte:body|window|document>` must appear at the template root. Adding a depth counter to `Builder` that increments inside `<element>`, `{#if}`, `{#each}`, `{#await}`, `{#key}` makes the validation a one-line check at emit time (`if b.nestDepth > 0 { fail }`). A pre-walk pass would have duplicated the recursion structure already in `emitChildren`. The defer-based decrement keeps the increment/decrement pairs visually adjacent so future `{#snippet}` and similar nestable constructs can join the count without rewiring control flow.
- **`AttrLet` is a parser-level kind, not a codegen heuristic.** Initially considered detecting `let:foo` by string-prefix in codegen alongside the existing classifyAttribute switch. That would have placed Svelte syntax knowledge in two layers. Adding `AttrLet` to the AST and routing `let:` through `classifyAttribute` keeps the parser the single source of attribute classification truth — codegen just consumes a typed `Kind`.
- **Slot-prop closure args type as `any` until the call site forces concrete types through Go's inference.** Component templates don't declare types; the producer's expression carries a Go type, the consumer's `let:foo` references it. Typing the closure as `func(item any, w *render.Writer)` lets the user cast or assert at the receiver site without forcing codegen to re-discover types from a separate Go AST pass. The cost is a missing compile-time error for `<List let:item>{item.NotAField}</List>` — `any` accepts anything; the user gets a runtime nil-pointer instead of a compile error. Acceptable trade because typed slot props would force a parallel Go type extraction pass tied to a non-existent file convention. Revisit when component imports formalize (#10).
- **The `liftSlotChildren` algorithm is intentionally one level deep.** Caller-side slot lifting (`<Card><h1 slot="header">...</h1>body</Card>` → two closures) walks immediate children only. Children of children carrying `slot=` are silently kept inside the parent slot — Svelte semantics don't bubble through nested elements. Tested by checking that `<Card><div><h1 slot="header"/></div></Card>` does NOT produce a Header slot binding (the `slot=` attribute is on a non-immediate child). The shallow walk is the right call: deep walking would conflate scope and cause silent miscompilation.
- **`<slot {item}/>` shorthand requires lexer-level mustache-in-tag support that we do not have.** First fixture pass used Svelte's shorthand syntax; the lexer rejects `{` in tag-attr position. Two paths: extend the lexer (large blast radius) or document longhand-only (`<slot item={item}/>`) for this PR. Chose longhand — the issue body's example uses both forms but the longhand is unambiguous. Filed nothing; shorthand support belongs to a future lexer extension issue if it surfaces.
- **`emitComponentCall` defers full props lifting to #10 by intent.** Phase 0bb owns slots; #10 owns component nesting + import resolution. The current emit assumes the user imports the component package under the lowercase-first-char alias (`Card` → `card`), constructs `card.CardProps{}` empty, and invokes `card.Card{}.Render(w, ctx, card.CardProps{}, _slots)`. That signature is the contract `GenerateComponent` produces. When #10 lands, the caller-side dispatch grows props composition; the slot-lifting half stays as-is.
- **Component fixtures must live OUTSIDE `testdata/codegen/` because `TestFixtures` walks that tree.** First placed component fixtures under `testdata/codegen/component/`; `TestFixtures` (in the FORBIDDEN `codegen_test.go`) tried to render them as page templates and failed on missing goldens. Moved to `testdata/component/` — same precedent as `testdata/golden/layout/` for layout fixtures. Treat `testdata/codegen/` as page-fixture-only; sibling subtrees own other generators.

### Self-rules

1. **When a new lowering shape mirrors an existing one (slots ↔ children closure), reuse the existing primitive instead of inventing parallel infrastructure.** A second runtime registry "for slots only" is a code smell — both compile-time decisions land on closure dispatch already. Same primitive = same audit surface = same bug class.
2. **Position validation that needs to know "am I at template root?" goes on `Builder` as a counter, not as a separate walk.** Increment on entering an enclosing construct, decrement on leaving (defer keeps the pair adjacent). One-line check at the validation site. Resists drift: any new nestable construct (`{#snippet}` next, then `{#await}` already covered) must remember to bump the counter to participate in root-only validation.
3. **Lowering for `<svelte:component this={X}>` calls `X.Render(w)` directly. Do not synthesize per-site interfaces.** The user's value already implements the receiver method; codegen's job is dispatch, not type wrapping. Reserve adapter generation for the day a real polymorphism case surfaces — until then, the indirection costs lines without buying anything.
4. **New attribute syntax extensions (`let:`, future `bind:` improvements) get an `AttrKind` value in the AST, not a string-prefix detection in codegen.** The parser is the single classification authority; codegen consumes typed kinds. Two-layer detection drifts the moment one layer needs to update.
5. **Receiver-side slot prop closures take `any` parameters; type recovery is the user's `let:` binding's job.** Codegen does not re-derive types from a separate Go pass for closure params. The runtime contract (positional args, last-arg `w`) is what makes the call site stable — type fidelity is a v1.0 perfectionist concern (#10 + future typed-props issue), not a Phase 0bb scope.
6. **When the issue body shows both shorthand and longhand forms of a Svelte syntax, test the longhand first against the existing lexer.** The shorthand may not parse. Confirmed earlier in Phase 0w with `{#await x then v}` shorthand; reconfirmed here with `<slot {item}/>` shorthand. The test corpus is the syntax oracle, not the issue body.
7. **Component-side fixtures live OUTSIDE `testdata/codegen/`.** That tree is owned by `TestFixtures` for page generation. New generators (component, layout, errorpage, ...) get sibling subtrees under `testdata/<kind>/` with their own walker in their own test file. Never extend `TestFixtures` to skip a new directory unless `codegen_test.go` is in scope — that file is FORBIDDEN in most concurrent phases.
