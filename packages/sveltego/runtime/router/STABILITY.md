# Stability — packages/sveltego/runtime/router

Last updated: 2026-04-30 · Version: pre-alpha

Tiers per [RFC #97](https://github.com/binsarjr/sveltego/issues/97).

## Tier legend

| Tier | Promise |
|---|---|
| `stable` | Won't break within the current major. Add-only changes; behavior changes documented in CHANGELOG. |
| `experimental` | May break in any minor release. Marked `// Experimental:` in godoc. |
| `deprecated` | Scheduled for removal. Marked `// Deprecated:` in godoc. Removed in next major. |

## Stable

- `router.Tree` — radix tree matching URL paths to routes
- `router.NewTree` — builds a Tree from a slice of Route values
- `router.Route` — route descriptor consumed by the server and emitted by codegen

## Experimental

- `router.Tree.Match` — matches a path and returns the Route, extracted params, and a hit boolean; signature may gain a context parameter
- `router.Tree.WithMatchers` — returns a new Tree with custom param matchers registered
- `router.Tree.Routes` — returns all routes registered in the tree
- `router.ParamMatcher` — interface for custom segment matchers
- `router.Matchers` — map of matcher name to ParamMatcher
- `router.MatcherFunc` — adapter that turns a plain function into a ParamMatcher
- `router.SegmentKind` — enumeration of segment kinds (static, param, optional, rest)
- `router.Segment` — parsed route segment (kind + name + optional matcher reference)
- `router.PageHandler` — handler type for page rendering
- `router.PageHeadHandler` — handler type for page `<head>` fragments
- `router.LayoutHandler` — handler type for layout rendering (accepts child render func)
- `router.LayoutHeadHandler` — handler type for layout `<head>` fragments
- `router.ErrorHandler` — handler type for error boundary rendering
- `router.LoadHandler` — handler type for page load functions
- `router.LayoutLoadHandler` — handler type for layout load functions
- `router.ActionsHandler` — handler type returning the actions map for a route
- `router.ServerHandlers` — map of HTTP method to handler for REST routes

## Breaking change procedure

Any `stable` symbol change requires an RFC issue describing the change and migration path, a PR that adds the new API alongside the old, at least one minor cycle with `// Deprecated:` godoc, and removal in the next major bump with a CHANGELOG entry. See [RFC #97](https://github.com/binsarjr/sveltego/issues/97) for the full procedure; CHANGELOG entries are generated by release-please.

## How to mark new symbols

Every exported symbol added in a PR **must** add a corresponding row to this file in the same PR. Place the row under `## Experimental` unless you have explicit approval to mark it `stable`. If the symbol is intentionally internal, place it under `## Internal-only`.
